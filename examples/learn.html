1.1

Why state machines

machine_perfect is a full-stack platform where state machines are the universal unit of definition, execution, and exchange. The same machine definition runs in the browser and on the server. The same expression language evaluates on both. Machines carry their own state, context, and routing requirements in the markup.

Why machines instead of arbitrary code? Because machines make impossible states impossible. Three booleans (loading, data, error) create eight combinations. Half are bugs: loading AND error, data AND loading AND error. You prevent them by remembering to check. You will forget.

A state machine has exactly the states you define. If there is no transition from loading to "loading + error", it cannot happen. Not because of validation. Because the transition does not exist.

Booleans (any language)

loaddataerr
---Idle
Y--Loading
-Y-Success
--YError
YY-Loading + data?
Y-YLoading + error?
-YYSuccess + error?
YYYAll three??

4 of 8 states are bugs you have to guard against.

State machine

idle
(after 1500 (to success)) loading
(animate) success
(animate) error

4 states. All valid. Cannot be loading AND error.

In machine_perfect, markup IS the state chart. States are elements. Transitions are attributes. If a transition does not exist in the markup, it cannot happen. This holds in the browser (HTML) and on the server (SCXML). The structure enforces the rules on every host.

1.2

Your first machine

Three attributes are all you need. mp creates a machine. mp-state declares a state. mp-to declares a transition. The first mp-state is the initial state.

<div mp="traffic">
  <div mp-state="red">
    <button mp-to="green">NEXT</button>
  </div>
  <div mp-state="green">
    <button mp-to="yellow">NEXT</button>
  </div>
  <div mp-state="yellow">
    <button mp-to="red">NEXT</button>
  </div>
</div>

Each state's content is created when the state is entered and destroyed when it is left. Only one state is active at a time. There is no JavaScript on this page.

The NEXT button in the red state transitions to green. There is no NEXT button that goes to yellow from red. That transition does not exist, so it cannot happen.

1.4

S-expression basics

An s-expression is a function call. Instead of f(a, b) you write (f a b). The first thing inside the parentheses is the function. Everything after it is an argument. That is the entire syntax.

JavaScript

add(2, 3)           // 5
count("hello")      // 5
"Hi " + name        // "Hi Ada"
x > 0 && y < 10     // true

S-expression

(+ 2 3)             ; 5
(count 'hello')      ; 5
(str 'Hi ' name)     ; "Hi Ada"
(and (> x 0) (< y 10)) ; true

Strings use single quotes (HTML attributes use double quotes, so single quotes avoid escaping). Numbers, true, false, and nil are written as-is. Bare words like name look up context values.

(+ x 3) = (+ x 3)

(str 'Hello ' name) = (str 'Hello ' name)

(> x 5) = (> x 5)

(if (> x 5) 'big' 'small') = (if (> x 5) 'big' 'small')

These expressions evaluate live against the context {"x": 7, "name": "Ada"}. Every mp-text, mp-show, mp-on, and mp-bind element accepts s-expressions.

2.1

Mutation

Functions that change data end with !. This is a convention, not decoration: the engine enforces it. set! sets a value. inc! increments. dec! decrements. toggle! flips a boolean.

(do expr1 expr2) runs multiple expressions in sequence. Use it when a transition needs to do more than one thing.

count

<button><mp-on event="click">(inc! count)</mp-on>+1</button>
<button>
  <mp-on event="click">(do (set! count 0) (to counting))</mp-on>
  Reset
</button>
  • (inc! count) mutates context. The ! means side effect
  • (do ...) sequences multiple expressions
  • (to counting) inside do transitions to a state
  • Mutation without (to) stays in the current state and re-renders

2.2

Guards

(when condition expr) blocks the expression from running unless the condition is true. If the condition is false, nothing happens. Use guards to prevent transitions that should not occur.

count

(= count 0) Nothing to reset.

(animate)

Reset to zero?

<button>
  <mp-on event="click">(when (!= count 0) (to confirm))</mp-on>
  Reset
</button>

Try clicking Reset when the count is 0. The guard blocks it. Increment first, then Reset works.

Guards are pure. They cannot use ! mutation functions. The engine enforces this at evaluation time.

mp-model creates a two-way binding between an input and a context value. Type in the input and the context updates. The context updates and the input reflects it.

mp-show shows or hides an element based on an expression. When the expression is falsy, the element gets hidden.

Attribute binding

<mp-bind attr="ATTR"> binds any HTML attribute to an expression. disabled, style, href, aria-label. The attribute updates when the expression's dependencies change.

<mp-class> adds CSS classes conditionally using (when condition 'class-name').

(str temp '°')
(cond (< temp 0) 'badge-info' (> temp 30) 'badge-error' true 'badge-success') (cond (< temp 0) 'Freezing' (> temp 30) 'Hot' true 'Comfortable') 
<div>
  <mp-class>(cond
    (< temp 0) 'badge-info'
    (> temp 30) 'badge-error'
    true 'badge-success')</mp-class>
</div>
<button>
  <mp-bind attr="disabled">(and
    (>= temp 0) (<= temp 30))</mp-bind>
</button>
  • mp-class with cond picks the first matching branch
  • <mp-bind attr="disabled"> disables the button when the condition is true
  • Drag the slider to see all three update reactively
You now know enough to build forms, interactive UIs, and multi-step workflows. Everything from here builds on these foundations.

3.1

Lists with mp-each

mp-each on a <template> element repeats the template for every item in an array. Inside the template, item properties are directly accessible as bare words. $index is the item's position.

mp-key gives each item a stable identity so the framework can add and remove items without re-rendering the whole list.

(= (count items) 0) No items yet.

(count items) items

<template mp-each="items" mp-key="name">
  <li>
    <span><mp-text>name</mp-text></span>
    <button><mp-on event="click">(remove-where! items :name name)</mp-on>x</button>
  </li>
</template>
  • Inside the template, name refers to the current item's name property
  • $index is the zero-based position in the array
  • (push! items (obj :name newItem)) adds an item
  • (remove-where! items :name name) removes the item whose name matches

3.2

Collection functions

The standard library has functions for filtering, sorting, mapping, and reducing arrays. These are used in mp-each, mp-text, and any other expression. They do not mutate the array.

(count (filter #(>= (get % :price) minPrice) items)) results

Total: £(reduce (fn [sum i] (+ sum (get i :price))) 0 (filter #(>= (get % :price) minPrice) items)) 

<mp-each>(filter #(>= (get % :price) minPrice) items)</mp-each>
(reduce (fn [sum i] (+ sum (get i :price))) 0 items)
  • #(expr) is shorthand for (fn [%] expr). % is the argument
  • (filter pred coll) returns items where the predicate is true
  • (reduce fn init coll) accumulates a result across the collection
  • (get item :price) reads a property. :price is a keyword (like a string key)
  • Also available: map, sort-by, group-by, find, first, last, take, count

Temporal behavior

<mp-temporal> on a state runs time-based expressions when the state is entered. (after ms expr) fires once after a delay. (every ms expr) fires repeatedly on an interval. Both are automatically cleaned up when the state is exited. No manual teardown.

(animate) applies CSS enter/leave transitions when switching states.

5

(do (every 1000 (dec! remaining)) (after 5000 (to done)))

remaining

(animate)

0

<div mp-state="running">
  <mp-temporal>(do
    (every 1000 (dec! remaining))
    (after 5000 (to done)))</mp-temporal>
  • Enter the running state: timer starts. Leave it: timer stops
  • No clearInterval, no useEffect cleanup, no lifecycle management
  • (animate) applies mp-enter-from / mp-leave-to CSS classes

6.1

Capability routing with mp-where

Everything so far runs in the browser. But some states need capabilities the browser does not have: database persistence, email notifications, external APIs. mp-where on a state declares what capabilities it requires.

When the machine transitions to a state with mp-where, the runtime checks whether the current host can satisfy those requirements. If not, it looks up the capability registry and routes the machine to a host that can. The machine's outerHTML (including current state and context) travels to the capable host. The host advances the machine and returns the result.

<div mp-state="draft">
  <!-- Form UI, browser handles this -->
</div>
<div mp-state="submitted"
     mp-where="(requires 'log' 'notify' 'persist')">
  <!-- Server handles this -->
</div>

The browser fills the form (draft state). On submit, the machine transitions to submitted. The browser cannot satisfy log, notify, and persist. The runtime sends the machine to a server that can. The server advances it through submitted, approved, fulfilled. The result HTML comes back to the browser.

The machine declares WHAT it needs. The host declares WHAT it provides. The framework mediates. No hardcoded URLs. No service discovery. The machine routes itself.

(animate)
6.2

URL routing with mp-url

mp-url on a state maps it to a browser URL. When the machine enters that state, the URL updates. When the user hits back/forward, the machine transitions to the matching state. Deep links work on page load.

<div mp-state="orders" mp-url="/orders">
<div mp-state="create" mp-url="/orders/new">
<div mp-state="detail" mp-url="(path '/orders/:id' orderId)">
  • Static paths: plain string. mp-url="/orders"
  • Parameterised paths: (path '/orders/:id' orderId) binds the :id URL segment to the context key orderId
  • States without mp-url do not affect the URL
  • mp-url travels in the markup. The server can parse it to derive SPA routes from the machine definition

6.3

Server pipelines

The server compiles HTML to SCXML and runs it through executePipeline. Effect adapters (log, notify, persist) are functions injected by the host. The machine definition is the single source of truth.

var machine = require('mp/machine');
var scxml = require('mp/scxml');
var def = scxml.compile(scxmlString);
var result = machine.executePipeline(def, {
  effects: {
    log: function(input) { console.log(input); },
    notify: function(input) { sendEmail(input); },
    persist: function(input) { db.save(input); }
  }
});

The pipeline tries each enabled transition in order. If a guard blocks one event, it tries the next. It stops at a final state, a route signal (needs capabilities this host does not have), or after maxSteps.

The purchase order app is 93 lines of server code. All it defines is what the host CAN DO (four effect adapters). The framework handles the rest.

(animate)
6.4

SCXML: the server surface

HTML is the browser authoring surface. SCXML is the server authoring surface. They compile to the same canonical format. The s-expression language is identical. htmlToScxml and scxmlToHtml convert between them.

HTML

<div mp="door">
  <div mp-state="locked">
    <button mp-to="unlocked">
  </div>
  <div mp-state="unlocked">
    <button mp-to="locked">
  </div>
</div>

SCXML

<scxml id="door">
  <state id="locked">
    <transition target="unlocked"/>
  </state>
  <state id="unlocked">
    <transition target="locked"/>
  </state>
</scxml>

Same machine. Two syntaxes. One engine. The browser writes HTML. The server can write SCXML. Both run the same guards, the same actions, the same effects. The machine is portable between hosts.
