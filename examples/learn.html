1.1

Why state machines

machine_native is a full-stack platform where state machines are the universal unit of definition, execution, and exchange. The same machine definition runs in the browser and on the server. The same expression language evaluates on both. Machines carry their own state, context, and routing requirements in the markup.

Why machines instead of arbitrary code? Because machines make impossible states impossible. Three booleans (loading, data, error) create eight combinations. Half are bugs: loading AND error, data AND loading AND error. You prevent them by remembering to check. You will forget.

A state machine has exactly the states you define. If there is no transition from loading to "loading + error", it cannot happen. Not because of validation. Because the transition does not exist.

Booleans (any language)

loaddataerr
---Idle
Y--Loading
-Y-Success
--YError
YY-Loading + data?
Y-YLoading + error?
-YYSuccess + error?
YYYAll three??

4 of 8 states are bugs you have to guard against.

State machine

idle
(after 1500 (to success)) loading
(animate) success
(animate) error

4 states. All valid. Cannot be loading AND error.

In machine_native, markup IS the state chart. States are elements. Transitions are attributes. If a transition does not exist in the markup, it cannot happen. This holds in the browser (HTML) and on the server (SCXML). The structure enforces the rules on every host.

1.2

Your first machine

Three attributes are all you need. mn creates a machine. mn-state declares a state. mn-to declares a transition. The first mn-state is the initial state (or use mn-initial to pick one).

<div mn="traffic">
  <div mn-state="red">
    <button mn-to="green">NEXT</button>
  </div>
  <div mn-state="green">
    <button mn-to="yellow">NEXT</button>
  </div>
  <div mn-state="yellow">
    <button mn-to="red">NEXT</button>
  </div>
</div>

Each state's content is created when the state is entered and destroyed when it is left. Only one state is active at a time. There is no JavaScript on this page.

The NEXT button in the red state transitions to green. There is no NEXT button that goes to yellow from red. That transition does not exist, so it cannot happen.

1.4

S-expression basics

An s-expression is a function call. Instead of f(a, b) you write (f a b). The first thing inside the parentheses is the function. Everything after it is an argument. That is the entire syntax.

JavaScript

add(2, 3)           // 5
count("hello")      // 5
"Hi " + name        // "Hi Ada"
x > 0 && y < 10     // true

S-expression

(+ 2 3)             ; 5
(count 'hello')      ; 5
(str 'Hi ' name)     ; "Hi Ada"
(and (> x 0) (< y 10)) ; true

Strings use single quotes (HTML attributes use double quotes, so single quotes avoid escaping). Numbers, true, false, and nil are written as-is. Bare words like name look up context values.

(+ x 3) = (+ x 3)

(str 'Hello ' name) = (str 'Hello ' name)

(> x 5) = (> x 5)

(if (> x 5) 'big' 'small') = (if (> x 5) 'big' 'small')

These expressions evaluate live against the context {"x": 7, "name": "Ada"}. Every mn-text, mn-show, mn-on, and mn-bind element accepts s-expressions.

2.1

Mutation

Functions that write data end with !. This is a convention the engine enforces. set! sets a value. inc! increments. dec! decrements. toggle! flips a boolean. push! appends to an array.

(do expr1 expr2) runs multiple expressions in sequence. Use it when a transition needs to do more than one thing.

count

<button><mn-on event="click">(inc! count)</mn-on>+1</button>
<button>
  <mn-on event="click">(do (set! count 0) (to counting))</mn-on>
  Reset
</button>
  • (inc! count) produces a new context with count incremented. The ! means write operation
  • (do ...) sequences multiple expressions
  • (to counting) inside do transitions to a state
  • A write without (to) stays in the current state and re-renders
(animate)
2.2

Guards

(when condition expr) blocks the expression from running unless the condition is true. If the condition is false, nothing happens. Use guards to prevent transitions that should not occur.

count

(= count 0) Nothing to reset.

(animate)

Reset to zero?

<button>
  <mn-on event="click">(when (!= count 0) (to confirm))</mn-on>
  Reset
</button>

Try clicking Reset when the count is 0. The guard blocks it. Increment first, then Reset works.

Guards are pure. They cannot use ! mutation functions. The engine enforces this at evaluation time.

Attribute binding

<mn-bind attr="ATTR"> binds any HTML attribute to an expression. disabled, style, href, aria-label. The attribute updates when the expression's dependencies change.

<mn-class> adds CSS classes conditionally using (when condition 'class-name').

(str temp '°')
(cond (< temp 0) 'badge-info' (> temp 30) 'badge-error' true 'badge-success') (cond (< temp 0) 'Freezing' (> temp 30) 'Hot' true 'Comfortable')

    (< temp 0) 'badge-info'
    (> temp 30) 'badge-error'
    true 'badge-success')</mn-class>
</div>
<button>
  <mn-bind attr="disabled">(and
    (>= temp 0) (<= temp 30))</mn-bind>
</button>
  • mn-class with cond picks the first matching branch
  • <mn-bind attr="disabled"> disables the button when the condition is true

2.5

Structured transitions

mn-to is a shorthand. For transitions with guards, actions, and emits, use the structured form: <mn-transition> with child elements for each concern.

Balance: £balance

(and (> amount 0) (<= amount balance)) (do (set! balance (- balance amount)) (set! amount 0))

(and (> amount 0) (> amount balance)) Insufficient funds. 

<button mn-to="withdraw">Withdraw</button>
<mn-transition event="withdraw" to=".">
  <mn-guard>(and (> amount 0) (<= amount balance))</mn-guard>
  <mn-action>(do (set! balance (- balance amount))
                  (set! amount 0))</mn-action>
</mn-transition>
  • event matches the mn-to value on the button
  • to="." is a self-transition — stay in the same state, re-render
  • <mn-guard> blocks the transition if false. Pure — cannot use ! functions
  • <mn-action> runs if the guard passes. Can use ! functions
  • You can also add <mn-emit>event-name</mn-emit> to fire an inter-machine event on transition

This is the same structure used in SCXML on the server. HTML and SCXML transitions have the same elements, same guards, same actions. One mental model across the entire stack.

Event modifiers

<mn-on event="name.modifier"> supports dot-separated modifiers after the event name. These control how the event is handled without extra logic in your expressions.

(do (set! submitted query) (set! query ''))

(not (empty? submitted)) Submitted: submitted 

<mn-on event="keydown.enter">(do
  (set! submitted query)
  (set! query ''))</mn-on>
ModifierEffect
click.outsideFires when the click is OUTSIDE the element (dismiss menus, close modals)
keydown.enterFires only on the Enter key
keydown.escapeFires only on the Escape key
click.preventCalls preventDefault() before running the expression

3.1

Lists with mn-each

mn-each on a <template> element repeats the template for every item in an array. Inside the template, item properties are directly accessible as bare words. $index is the item's position.

mn-key gives each item a stable identity so the framework can add and remove items without re-rendering the whole list.

(= (count items) 0) No items yet.

(count items) items

<template mn-each="items" mn-key="name">
  <li>
    <span><mn-text>name</mn-text></span>
    <button><mn-on event="click">(remove-where! items :name name)</mn-on>x</button>
  </li>
</template>
  • Inside the template, name refers to the current item's name property
  • $index is the zero-based position in the array
  • (push! items (obj :name newItem)) adds an item
  • (remove-where! items :name name) removes the item whose name matches
(animate)
3.2

Collection functions

The standard library has functions for filtering, sorting, mapping, and reducing arrays. These are used in mn-each, mn-text, and any other expression. They do not mutate the array.

(count (filter #(>= (get % :price) minPrice) items)) results

Total: £(reduce (fn [sum i] (+ sum (get i :price))) 0 (filter #(>= (get % :price) minPrice) items)) 

<mn-each>(filter #(>= (get % :price) minPrice) items)</mn-each>
(reduce (fn [sum i] (+ sum (get i :price))) 0 items)
  • #(expr) is shorthand for (fn [%] expr). % is the argument
  • (filter pred coll) returns items where the predicate is true
  • (reduce fn init coll) accumulates a result across the collection
  • (get item :price) reads a property. :price is a keyword (like a string key)
  • Also available: map, sort-by, group-by, find, first, last, take, count
(animate)

Temporal behavior

<mn-temporal> on a state runs time-based expressions when the state is entered. (after ms expr) fires once after a delay. (every ms expr) fires repeatedly on an interval. Both are automatically cleaned up when the state is exited. No manual teardown.

(animate) applies CSS enter/leave transitions when switching states.

4.6

External components with mn-import

Component templates can live in separate files. <link rel="mn-import"> loads them before the app boots. The imported file contains <template mn-define="name"> elements — identical to inline definitions.

<!-- In your HTML head or body -->
<link rel="mn-import" href="components/card.html" />
<link rel="mn-import" href="components/modal.html" />
<!-- components/card.html contains: -->
<template mn-define="my-card">
  <div mn-state="front">...</div>
  <div mn-state="back">...</div>
</template>
<!-- Then use it anywhere: -->
<div mn="my-card" mn-ctx='{"title":"Hello"}'></div>
  • Imports load in parallel during boot, before any machines are created
  • One file can contain multiple <template mn-define> elements
  • The import order does not matter — all imports resolve before init runs
  • This keeps your main HTML clean and your components reusable across pages

4.7

Decomposition

When a machine transitions between states, the previous state's DOM is destroyed and the new state's DOM is created. Child machines inside a state are destroyed and recreated on every parent transition. This is expensive and unnecessary if the child machine hasn't changed.

The fix: put shared elements at the template root — outside any mn-state. Root elements persist across state transitions. Only the state content swaps.

<template mn-define="app">
  <!-- Shared navbar — outside states, never re-renders -->
  <nav>...buttons...</nav>
  <!-- Only the content area changes -->
  <div mn-state="page-a">...</div>
  <div mn-state="page-b">...</div>
</template>

Machines that share data use $store. Machines that trigger transitions use emit / mn-receive. Shared UI reads $store — dep tracking re-evaluates only the changed bindings. No cascade.

<!-- Navbar reads $store.session for user info -->
<span><mn-text>(get $store.session.user :name)</mn-text></span>
<!-- Sign out button emits to another machine -->
<button><mn-on event="click">(emit logout)</mn-on>Sign out</button>
  • Template root elements persist across state transitions
  • Machine boundaries are rendering boundaries — a child machine only updates when ITS data changes
  • $store for shared data, emit/receive for events — no nesting required
  • Small focused machines decompose naturally. The performance model IS the architecture model

5.5

Initial state with mn-initial

By default, the first mn-state element is the initial state. mn-initial="stateName" on the machine element overrides this — the machine starts at the named state instead.

This is critical for resuming machines. A machine stored at "approved" can be re-created with mn-initial="approved" and it starts right there, with all its context intact.

<!-- Starts at 'draft' (first state) -->
<div mn="workflow" mn-ctx='{"title":"..."}'>
  <div mn-state="draft">...</div>
  <div mn-state="submitted">...</div>
  <div mn-state="approved">...</div>
</div>
<!-- Starts at 'approved' directly -->
<div mn="workflow" mn-initial="approved"
     mn-ctx='{"title":"...","approved_at":"..."}'>
</div>
  • The machine definition still has all states and transitions — only the starting point changes
  • Combined with the server's <invoke>, this is how stored machines are loaded as live children at their persisted state
  • This learn app uses it: <div mn="learn" mn-initial="L1-1">

6.1

Immutable context

Despite the ! suffix, mutation functions do not modify context in place. (set! x 5) creates a new context object where x is 5. The original context is untouched. The framework threads the new context through every step of a transition.

// What happens inside the engine:
var result = engine.exec("(set! x 5)", ctx);
// result.context is a NEW object: { x: 5, ... }
// ctx is UNCHANGED
// The framework threads it:
// exit hook   → new ctx₁
// action      → new ctx₂ (from ctx₁)
// entry hook  → new ctx₃ (from ctx₂)
// inst.ctx = ctx₃  (single assignment, never mutation)

This applies everywhere:

  • Engine: all ! functions write to a scope. newContext() merges scope into a new object
  • Canonical machine: sendEvent threads new context through exit → action → init
  • Browser: every event handler produces a new context via the same path
  • $store: writes replace the entire store object and broadcast immutably to all machines
  • Adapters: return data. The framework merges it. Adapters never receive a mutable reference

Nested paths like items.0.qty use structural sharing — only the path from root to the changed leaf is copied. Everything else is shared by reference.

(animate)
6.2

Implicit error state

Every machine gets an implicit error state (final) injected automatically if none is defined. When any guard, action, or effect throws during a transition, the machine transitions to error with two context keys set:

// On throw during sendEvent:
context.$error       = "Cannot read property 'x' of undefined"
context.$errorSource = "submit"  // the event that caused it
state                = "error"   // machine stops here (final)

No machine silently swallows exceptions. The error state is a dead end — the machine stops, the error is visible in context.

If you want recovery, define your own error state with outbound transitions:

<div mn-state="error">
  <p><mn-text>$error</mn-text></p>
  <button mn-to="retry">Try again</button>
</div>
  • The implicit error state is skipped by validate() unreachability checks
  • On the server, pipeline errors transition to error the same way — it is not a browser-only feature
  • The framework provides the safety net. Error handling logic is machine authoring, not framework magic
(animate)
6.3

Error recovery

The implicit error state is final — the machine stops. If you want recovery, define your own error state with outbound transitions. The framework uses yours instead of injecting the implicit one.

Implicit (default)

<!-- No error state defined -->
<!-- Framework injects: -->
<final id="error"/>
<!-- Machine stops. No recovery. -->

Explicit (author-defined)

<state id="error">
  <transition event="retry"
    target="loading"/>
</state>
<!-- Machine recovers. -->

When a guard, action, or effect throws, the framework sets $error (the message) and $errorSource (the event that caused it) in context, then transitions to error. Your template renders them:

<div mn-state="error">
  <p><mn-text>$error</mn-text></p>
  <p>Source: <mn-text>$errorSource</mn-text></p>
  <button mn-to="retry">Try again</button>
</div>
  • The framework provides the safety net (implicit). The author provides the recovery strategy (explicit)
  • An explicit error state is a regular state — it can have guards, actions, timers, anything
  • If your error state is also final, it behaves like the implicit one but with your template
  • This works on both browser and server. Pipeline errors transition to the error state the same way
(animate)
7.1

Capability routing with mn-where

Everything so far runs in the browser. But some states need capabilities the browser does not have: database persistence, email notifications, external APIs. mn-where on a state declares what capabilities it requires.

When the machine transitions to a state with mn-where, the runtime checks whether the current host can satisfy those requirements. If not, it looks up the capability registry and routes the machine as SCXML to a host that can. The machine carries its own state, context, transitions, and routing requirements. The host advances it and pushes the result back.

<div mn-state="draft">
  <!-- Form UI, browser handles this -->
</div>
<div mn-state="submitted">
  <mn-where>(requires 'log' 'notify' 'persist')</mn-where>
  <!-- Server handles this -->
</div>

The browser fills the form (draft state). On submit, the machine transitions to submitted. The browser cannot satisfy log, notify, and persist. The runtime sends the machine to a server that can.

The machine declares WHAT it needs. The host declares WHAT it provides. The framework mediates. No hardcoded URLs. No service discovery. The machine routes itself.

7.2

URL routing with mn-url

mn-url on a state maps it to a browser URL. When the machine enters that state, the URL updates. When the user hits back/forward, the machine transitions to the matching state. Deep links work on page load.

<div mn-state="list" mn-url="/items">
<div mn-state="create" mn-url="/items/new">
<div mn-state="detail" mn-url="(path '/items/:id' itemId)">
  • Static paths: plain string. mn-url="/items"
  • Parameterised paths: (path '/items/:id' itemId) binds the :id URL segment to the context key itemId
  • States without mn-url do not affect the URL
  • mn-url travels in the markup. The server can parse it to derive SPA routes from the machine definition

7.3

SCXML: the server surface

HTML is the browser authoring surface. SCXML is the server authoring surface. Both compile to the same canonical machine definition. The s-expression language is identical. There is no conversion between them — they are two syntaxes for the same thing.

HTML (browser)

<div mn="door">
  <div mn-state="locked">
    <mn-transition event="unlock" to="unlocked">
      <mn-guard>(= code '1234')</mn-guard>
    </mn-transition>
  </div>
  <div mn-state="unlocked" />
</div>

SCXML (server)

<scxml name="door">
  <state id="locked">
    <transition event="unlock" target="unlocked"
      cond="(= code '1234')" />
  </state>
  <state id="unlocked" />
</scxml>

Same machine. Two syntaxes. One engine. SCXML is the wire format — it is what travels between hosts, what gets stored in the database, and what gets validated with XML tooling. HTML only exists on the browser node.

SCXML uses mn: namespace extensions for machine_native features: <mn:where>, <mn:temporal>, mn-action, mn-emit. Standard SCXML attributes (cond, target, event) work as-is.

7.4

Server pipelines

When a machine arrives at a server host, the server compiles its SCXML, creates an instance, and runs executePipeline. The pipeline advances the machine through transitions automatically until it reaches a final state, a blocking mn:where, or a step limit. Effect adapters fire along the way.

var machine = require('mn/machine');
var scxml = require('mn/scxml');
var def = scxml.compile(incomingScxml);
var result = machine.executePipeline(def, {
  effects: {
    log:     function(input) { console.log(input); },
    notify:  function(input) { sendEmail(input); },
    persist: function(input) { db.save(input); }
  }
});
  • The server has zero business logic. It wires up adapters and calls executePipeline
  • The machine's guards and transitions decide what happens — not the server
  • If a state has mn:where and the server lacks the required capabilities, the pipeline stops and returns a route signal
  • The pipeline tries each enabled transition. If a guard blocks, it tries the next event. It stops at a final state or after maxSteps

7.5

Effects with invoke!

(invoke! :type 'name' :input data) declares an effect for the host to execute. The machine says WHAT should happen. The host's effect adapter decides HOW. This is how machines trigger persistence, notifications, logging, and any other side effect.

<!-- In SCXML: -->
<transition event="submit" target="submitted">
  <mn-action>(do
    (set! submitted_at (now))
    (invoke! :type 'log' :input title)
    (invoke! :type 'persist'))</mn-action>
</transition>
  • :type names the effect adapter: 'persist', 'notify', 'log', 'fulfil', or any custom name
  • :input passes data to the adapter. Can be any expression
  • Effects are collected during action evaluation and dispatched by the host after the transition completes
  • The persist effect automatically receives the machine's current SCXML snapshot — the machine IS the persistence format
  • Effect types listed in mn:where determine routing. (requires 'persist') routes to a host with a persist adapter

In the browser, MachineNative.fn() serves a similar purpose — it registers JavaScript functions callable from s-expressions. On the server, effect adapters are the equivalent.

7.6

Fire-and-forget transport

When mn-where routes a machine to a remote host, the browser does not wait for the result. It sends the machine as SCXML via HTTP POST and receives a 202 immediately. The UI stays responsive.

Results come back via Server-Sent Events (SSE). The browser opens an EventSource connection to its server on init, identified by a session ID. When the server finishes pipeline execution, it pushes the result SCXML through that channel. The browser decodes it, recompiles the definition, and updates the machine instance.

// Browser → Server (fire and forget)
POST /api/machine
Content-Type: application/xml
X-MN-Session: browser-abc
Body: <scxml name="workflow" ...>...</scxml>
→ 202 Accepted (immediate)
// Server → Browser (push when done)
GET /sse/browser-abc
event: machine
data: base64(result SCXML)
  • This is uniform — data loads, approval workflows, and pipeline execution all use the same pattern
  • There is no separate request/response path for "simple" vs "complex" operations
  • The transport type is declared at host registration. HTTP POST is one type. MQTT, WebSocket, and others can be added as transport adapters
  • The registry is a phone book: it tells you where to send machines and how. It does not relay them
(animate)
7.7

Machine-as-persistence

Machines are stored as complete SCXML snapshots. The SCXML carries state, context, transitions, guards, and effects — everything needed to resume execution. There is no separate "data model" extracted from the machine. The machine document IS the persistence format.

<invoke type="scxml" src="machine-name"/> on a state loads stored machines as live children. The pipeline's invoke resolver retrieves SCXML snapshots from persistence, embeds them in the response, and the browser creates live child machine instances from each one.

<!-- In SCXML: the 'list' state loads stored workflows -->
<state id="list">
  <invoke type="scxml" src="workflow"/>
</state>
<!-- Filter by state: only load blocked ones -->
<state id="review-queue">
  <invoke type="scxml" src="workflow" mn-state="needs-review"/>
</state>

The invoke resolver also computes _invokeCounts: { total, byState: { ... } }. This is merged into the parent's context so it can display aggregate state (e.g. "3 completed, 1 pending review").

In the browser, _stampAndEnter creates a child machine DOM element for each invoked machine:

<!-- Created automatically from invoke content: -->
<div mn="workflow" mn-initial="completed"
     mn-ctx='{"title":"...","completed_at":"..."}'>
</div>

Each child renders itself using its paired .mn.html template. The parent does not need to know how children look — it just invokes them and they take care of themselves.

7.8

Context projection (mn:project)

When stored machines are loaded as invoke children and sent to the browser, the full context travels in the SCXML. But the browser is untrusted — sensitive fields (internal notes, cost breakdowns, margin percentages) would be visible in devtools even if the UI hides them.

mn:project is an s-expression on the machine's SCXML root. It declares what context travels to different audiences. The projection runs on the server during invoke resolution. Stripped fields never leave the server — not hidden, never sent.

<scxml name="purchase-order" ...>
  <mn:project when="(!= (get $user :role) 'director')">
    (obj
      :title title
      :amount amount
      :item_count (count items))
  </mn:project>
  <state id="draft">...</state>
</scxml>

The when condition evaluates against the parent machine's context — who's asking. The body evaluates against the child machine's context — the stored data. First matching when wins. No match = full context travels.

  • The body is a full s-expression. It can compute, reshape, aggregate — (count items) gives a count without sending the items
  • $state is available — the canonical machine's stored state
  • Non-directors get {title, amount, item_count}. Directors get everything. The machine defines its own data boundary
  • This is the third axis of machine self-description: mn:where (where), mn:guard (when), mn:project (what)
(animate)
7.9

Machine transforms (mn:project as=)

A projected context on the same machine is a crippled machine — guards and actions reference fields that were stripped. as solves this by deriving a completely different machine.

<mn:project as="purchase-order-status"
  when="(!= (get $user :role) 'director')">
  (obj
    :title title
    :amount amount
    :item_count (count items)
    :$initial (cond
      (= $state 'fulfilled') 'fulfilled'
      (= $state 'rejected') 'rejected'
      true 'pending'))
</mn:project>

The as attribute names the target machine. The invoke resolver builds a minimal SCXML envelope with the derived machine name, the projected context, and $initial as the starting state. The browser looks up the template by the derived name. Both machines are complete and independently functioning.

Reverse path: the derived machine can act. If Alice cancels her order, the status machine routes to the server with $canonical_id in context. The server loads the canonical machine by ID, merges Alice's edits, runs the canonical through the pipeline, and re-projects the result back as the derived machine.

<!-- Forward: canonical → derived -->
Full PO (notes, margin, items) → Status card (title, amount, count)
<!-- Reverse: derived → canonical → re-project -->
Status sends cancel + $canonical_id → Server loads full PO →
Applies cancel → Re-projects → Status card at 'cancelled'
  • The canonical machine defines both directions. The server executes mechanically
  • Sensitive fields never reach the browser. Not hidden — never sent
  • The derived machine has its own SCXML definition, its own states, its own template
  • This is how auth works in MN: the machine declares its own security model. No middleware. No server changes
(animate)
7.10

Escape hatch (MachineNative.fn)

Sometimes you need JavaScript. A chart library. A file upload. A camera API. MachineNative.fn() registers a JavaScript function callable from s-expressions. The machine carries the data. The function does the rendering.

// Register once, use from any machine
MachineNative.fn('renderChart', function(canvas, type, data) {
  return new Chart(canvas, { type: type, data: data });
});
<!-- Call from mn-init with $refs and context data -->
<mn-init>(renderChart $refs.chart 'doughnut'
  (obj :labels (list 'Done' 'Pending')
       :data (list fulfilled pending)))</mn-init>
  • $refs gives access to DOM elements via mn-ref
  • Context data drives the chart — reactive by default
  • Works with any JS library: Chart.js, D3, Mapbox, video players
  • The escape hatch is for rendering and browser APIs — business logic stays in the machine
  • On the server, engine.fn() serves the same purpose for server-side functions

That is the complete framework.

One shared engine. Two host runtimes. S-expressions as the single expression language. Machines as the universal unit of definition, execution, transport, and persistence. Capability-based routing. Immutable context. Context projection. Machine transforms. Fire-and-forget transport. Machine-as-persistence. Third-party integration via escape hatch.
